## Overview

`FlintClient` extends `@fluxerjs/core`'s `Client` class and serves as the foundation of your bot. It provides getters for your registered handlers and a `login` method that ties everything together.

You never instantiate `FlintClient` directly, instead you extend it with your own class and register the handlers you need.

## Creating a client
```ts src/index.ts
import { FlintClient, CommandHandler, ListenerHandler } from "@flint.js/core"

class MyClient extends FlintClient {
    commandHandler: CommandHandler
    listenerHandler: ListenerHandler

    constructor() {
        super()

        this.listenerHandler = new ListenerHandler(this, {
            directory: "./src/listeners"
        })

        this.commandHandler = new CommandHandler(this, {
            directory: "./src/commands",
            prefix: "!"
        })

        this.commandHandler
            .useListenerHandler(this.listenerHandler)
    }

    override async login(token: string) {
        await this.listenerHandler.loadAll()
        await this.commandHandler.loadAll()
        return super.login(token)
    }
}

const client = new MyClient()
await client.login(process.env.TOKEN as string)
```

## Handler registration order

The order you call `loadAll()` in matters. Handlers that others depend on should be loaded first:
```ts
// inhibitors and listeners first, commandHandler depends on both
await this.inhibitorHandler.loadAll()
await this.listenerHandler.loadAll()
await this.commandHandler.loadAll()
```

## Available handlers

| Handler | Description |
|---|---|
| `commandHandler` | Loads and manages commands |
| `listenerHandler` | Loads and manages listeners |
| `inhibitorHandler` | Loads and manages inhibitors |
| `monitorHandler` | Loads and manages monitors |

All handlers are optional. Only register the ones you need. Accessing an unregistered handler will throw a helpful error at runtime.

## Client options

`FlintClient` accepts a `FlintClientOptions` object in its constructor, which is passed through to `@fluxerjs/core`'s `Client`:
```ts
super({
    intents: 0,
    presence: {
        status: StatusType.ONLINE,
        custom_status: {
            text: "Built with @flint.js/core"
        }
    }
})
```

## Adding a logger

If you have `@flint.js/logger` installed, you can attach it to the client for consistent styled output across your whole bot:
```ts
import { Logger } from "@flint.js/logger"

class MyClient extends FlintClient {
    constructor() {
        super()
        this.logger = new Logger({ prefix: "MyBot" })
        // ...
    }
}
```

## Extending the client

Since you control the client class, you can add anything you need directly to it. Database connections, utility methods, shared state:
```ts
import { PrismaClient } from "@prisma/client"

class MyClient extends FlintClient {
    db = new PrismaClient()
    commandHandler: CommandHandler
    listenerHandler: ListenerHandler

    // ...
}
```

Your commands and listeners receive the client as their first argument, so everything you add here is accessible wherever you need it.

<Tip>
    If you add custom properties to your client, consider creating a type alias or casting `client` in your commands to get full type safety on your custom properties.
</Tip>
