## Overview

Flint currently ships with three built-in inhibitors that cover some common use cases. They are not enabled by default, you opt in by passing them as `builtins` to your `InhibitorHandler`.
```ts
import {
    InhibitorHandler,
    ChannelTypeInhibitor,
    UserPermissionsInhibitor,
    BotPermissionsInhibitor,
} from "@flint.js/core"

this.inhibitorHandler = new InhibitorHandler(this, {
    directory: "./src/inhibitors",
    builtins: [
        new ChannelTypeInhibitor()
        new UserPermissionsInhibitor()
        new BotPermissionsInhibitor()
    ]
})
```

## Inhibitors

The following inhibitors are built-in to `@flint.js/core`

### ChannelType

Blocks a command from running if it is used outside of its `allowedChannels` list. If `allowedChannels` is empty or not set, the command is allowed in all channels.
```ts
export default defineCommand({
    name: "ping",
    description: "Pong!",
    allowedChannels: ["123456789012345678"],

    async execute(client, message) {
        await message.reply("Pong!")
    }
})
```

When blocked, the `CommandDenied` event is emitted with `reason: "channel"`.

### UserPermissions

Blocks a command from running if the user is missing any of the required permissions defined in the command's `permissions` array.
```ts
import { PermissionFlags } from "@fluxerjs/core"

export default defineCommand({
    name: "ban",
    description: "Ban a member",
    permissions: [PermissionFlags.BanMembers],

    async execute(client, message, args) {
        // only runs if the user has BanMembers permission
    }
})
```

When blocked, the `CommandDenied` event is emitted with `reason: "user.permissions"` and a `missing` array of the permissions the user is lacking.

### BotPermissions

Works the same as `UserPermissions` but checks the bot's own permissions instead of the user's.
```ts
import { PermissionFlags } from "@fluxerjs/core"

export default defineCommand({
    name: "ban",
    description: "Ban a member",
    permissions: [PermissionFlags.BanMembers],

    async execute(client, message, args) {
        // only runs if the bot has BanMembers permission
    }
})
```

When blocked, the `CommandDenied` event is emitted with `reason: "bot.permissions"` and a `missing` array of the permissions the bot is lacking.

<Note>
    `UserPermissions` and `BotPermissions` both read from the same `permissions` array on the command. You only need to define the permissions once.
</Note>

## Inhibitor results

Each built-in inhibitor emits a specific reason when it blocks a command. Use these in your `CommandDenied` listener to send appropriate feedback:

| Inhibitor | Reason | Extra fields |
|---|---|---|
| `ChannelType` | `"channel"` | None |
| `UserPermissions` | `"user.permissions"` | `missing: PermissionResolvable[]` |
| `BotPermissions` | `"bot.permissions"` | `missing: PermissionResolvable[]` |
