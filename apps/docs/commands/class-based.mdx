## Overview

For more advanced use cases, you can define commands by extending the `Command` class directly instead of using `defineCommand`. Both approaches produce the same result. Use whichever feels more natural to you.
```ts src/commands/ping.ts
import type { FlintClient } from "@flint.js/core"
import type { Message } from "@fluxerjs/core"
import { Command } from "@flint.js/core"

export default class PingCommand extends Command {
    constructor() {
        super("ping", {
            description: "Pong!"
        })
    }

    async execute(client: FlintClient, message: Message) {
        await message.reply("Pong! 🏓")
    }
}
```

<Tip>
    If your client extends `FlintClient` with custom properties or handlers, annotate `client` directly in `execute` as your custom client that extends `FlintClient`
</Tip>

## Constructor

The `Command` constructor takes a name and an options object:
```ts
super("commandName", {
    description: "A short description",
    category: "Misc",
    aliases: ["alias1", "alias2"],
    cooldown: 5000,
    permissions: [PermissionFlags.BanMembers],
    allowedChannels: ["123456789"],
    prefixes: [">>"],
    disabled: false,
    args: [
        { id: "member", type: "member" },
        { id: "reason", type: "string", match: "rest" }
    ] as const
})
```

All options are the same as `defineCommand`. See the [command options](/commands/defining-commands#command-options) reference for the full list.

## When to use classes

The factory approach with `defineCommand` is recommended for most commands. Classes are better suited for situations where you need:

- Shared logic between commands via inheritance
- Private class fields for state (e.g. cooldown maps, caches)
- A more structured style across a larger codebase
```ts src/commands/ban.ts
import { Command } from "@flint.js/core"
import type { FlintClient } from "@flint.js/core"
import type { Message } from "@fluxerjs/core"
import { PermissionFlags } from "@fluxerjs/core"

export default class BanCommand extends Command {

    #auditLog = new Map<string, string[]>()

    constructor() {
        super("ban", {
            description: "Ban a member from the server",
            category: "Moderation",
            aliases: ["boot"],
            permissions: [PermissionFlags.BanMembers],
            args: [
                { id: "member", type: "member" },
                { id: "reason", type: "string", match: "rest", default: "No reason provided" }
            ] as const
        })
    }

    async execute(client: FlintClient, message: Message, args: { member: GuildMember | null, reason: string }) {
        if (!args.member) {
            return message.reply("Please provide a member to ban.")
        }

        this.#auditLog.set(args.member.id, [
            ...(this.#auditLog.get(args.member.id) ?? []),
            args.reason
        ])

        await args.member.ban()
        await message.reply(`${args.member.user.username} has been banned. Reason: ${args.reason}`)
    }
}
```

<Note>
    Unlike `defineCommand`, class-based commands require you to manually annotate the `args` parameter type in `execute` since TypeScript cannot infer it through class inheritance the same way it can through generics.
</Note>
